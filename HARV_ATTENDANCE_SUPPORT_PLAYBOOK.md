# harv-attendance — support playbook

Written during development, not after launch — every feature that ships, or bug that gets fixed, gets a row here the same sprint, per the Faction Group 3-tier model (source video: facebook.com/reel/1023018453988059, watched 2026-07-10 — see design-system vault `wiki/log.md` 2026-07-10).

**Standing rule: this file gets updated every time anything project-relevant happens in this repo — a feature ships, a bug is fixed, a file gets moved, or a decision/strategy call gets made — not batched, not deferred.** (Scope generalized 2026-07-14 — see global `CLAUDE.md`'s "Support playbooks" rule.) The point is to know later exactly what was built, what broke, what was decided and why, without having to reconstruct it from commit messages, git history, or chat memory.

## The 3 tiers

1. **Automated resolution** — known issue, documented fix, no human needed.
2. **Assisted triage** — unknown issue, agent gathers context (logs, recent commits, error) and escalates to Hazem *with* that context, not a bare alert.
3. **Incident response** — security or data-integrity issue, human-led, agent's job is to surface it loud and fast.

**Current state (2026-07-10): documentation, plus a real regression test suite (`npm test`) added same day.** No log-piping or auto-resolution wired up yet — there are no production users, so there's no real traffic to justify that infra. The test suite is a different thing from tier-1 auto-resolve (it catches regressions of known-fixed bugs before they ship, not production incidents in real time) — revisit tier-1 automation once real usage surfaces real recurring issues (see TO DO.md).

## Security hardening from The Faction Group reels (2026-07-17)

Source: three Facebook reels from The Faction Group LLC (see design-system vault's `Atlas/Faction Group - Dev Notes.md` for the full source material). Two prior videos from the same creator already drove standing rules here (AI-review merge gate, this playbook itself) — see the top of this file and `feedback_ai_pr_review_default` in Claude's memory.

**Set up same session, all free/reversible (public repo):**
- GitHub secret scanning + push protection — enabled via repo settings. Blocks a commit containing an obvious API key/token before it ever lands, and flags anything already in history.
- Dependabot security updates — enabled. Auto-PRs a fix when an installed npm package gets a known CVE.
- `.github/dependabot.yml` — weekly dependency-update PRs (separate from security-only updates above).
- `.github/workflows/codeql.yml` — GitHub's own static code scanner (CodeQL), runs on every push/PR to `master` plus weekly. This is the "GitHub's built-in code scanning" option the reel named as a free alternative to Snyk.
- `.github/workflows/zap-baseline.yml` — OWASP ZAP in **passive/baseline mode** (crawls + inspects responses, never attempts SQLi/XSS/auth-bypass payloads) against the live public routes at `attendance.harvcentereg.com`, weekly + manual trigger. Runs against production because no staging environment exists — safe because baseline mode is read-only. `/admin*` routes sit behind Cloudflare Access, which ZAP can't authenticate through unassisted, so this scan only covers what's actually publicly reachable (`/`, `/register`, `/scan`, `/student`, `/public/roster`) — which is the real external attack surface anyway.

**Deliberately not automated — Burp Suite equivalent (manual checklist, for whoever next does a security pass):** Burp is a paid GUI tool for intercepting/replaying your own requests; it can't be wired into CI the way ZAP can. Given this app already has 28+ PRs of `claude-review` coverage that already found and fixed real IDOR/auth-bypass/XSS/money bugs (see Known issues below), a manual Burp pass has lower marginal value here than on a less-reviewed app — but if one is ever done, check specifically:
1. Try any `/admin/*` GET/POST with the `Cf-Access-Jwt-Assertion` header stripped (should 403 — already covered by a regression test, worth re-verifying live).
2. As a logged-in `clerk` or `viewer` Access session, try every `owner`-only POST route directly (settlement, staff role changes, ledger) — should 403.
3. Try `/scan?student=<id>` and `/admin/scan-mark` with an inactive/invalid `group_id` — should reject (this exact class of bug was found+fixed, PR #8; confirm it's still enforced after any later refactor).
4. Try `/admin/students/:id/photo` and the teacher-photo equivalent without the Access header — should 403 at the entrypoint check, not just render empty.
5. Fuzz the photo upload (`/admin/teachers/new`, `/admin/teachers/:id/edit`) with a renamed non-image file — `ALLOWED_PHOTO_TYPES` checks the browser-reported MIME type, which a raw HTTP client can spoof; worth confirming the stored bytes are actually validated, not just trusted from the `Content-Type` the client sent.

## Photo storage — plain-language explainer (deferred, for review, not yet acted on)

**Written 2026-07-17 for a non-coder reading this later — no jargon assumed.**

**What's happening right now:** every student and teacher photo you or a clerk uploads gets saved *inside the same database* that also holds names, phone numbers, attendance records, and payment history. Technically: the photo's raw bytes go straight into a database column (`schema.sql:9,59` — `photo BLOB`/`photo_blob BLOB`), the same way a phone number or a date would.

**Why that's the wrong drawer for it:** think of the database like a small filing cabinet built for paperwork — forms, numbers, short text. It's fast and cheap *because* everything in it is small. A photo is a completely different kind of thing — much bigger, and not something you ever need to search through or calculate with the way you search a name or sum a payment. Every time the app opens that filing cabinet to grab something small (say, checking if a student is marked present today), it's now also carrying around every photo stored nearby, even though it didn't ask for them. As more photos pile in, every backup of the database gets bigger and slower, and the database service itself (which Cloudflare prices and optimizes for small structured data, not files) gets more expensive to run — this is the literal "$200/month database becomes an $800/month database" line from the reel that flagged this pattern.

**What the fix would look like:** move photos into a separate service built specifically for files — Cloudflare's own "R2" storage (same company already running this app, so it plugs into the existing setup with no new vendor). Photos would be served through a CDN too, meaning they'd actually load *faster* for staff, not slower. The database would then just keep a short label — "photo for this student lives at address #57" — instead of the photo itself. This is what the reel meant by "the database stores a URL, not the file."

**Why this isn't urgent right now:** the app currently has a small number of real students/teachers, so the actual cost/speed hit today is small — this is a "will get worse as it grows," not a "broken now," situation.

**What doing the fix would actually involve (so the size of the task is clear before deciding to schedule it):**
1. Create a new R2 storage bucket (a few minutes, effectively free at this data size — R2's free tier is 10GB, far more than these photos will ever total).
2. Write a one-time script to copy every existing photo out of the database and into that bucket.
3. Change the handful of places in the code that currently save a new photo (teacher editor, student registration) to write to R2 instead of the database.
4. Change the two places that currently serve a photo back out (`/admin/students/:id/photo` and the teacher-photo equivalent) to fetch from R2 instead of the database.
5. Test thoroughly before deploying, since this touches photos already live in production for real students and teachers — a mistake here means someone's photo silently disappearing, not just a cosmetic bug.

None of this has been started. If/when it's picked up, it deserves its own dedicated session rather than being folded into other work, given it touches live production data. Cross-reference: `TO DO.md`'s harv-attendance section already separately flagged R2 as the right fit for a *different* future feature (course-content storage) — same tool, two independent uses.
